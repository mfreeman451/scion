# Harness-Agnostic Templates: Feasibility Research

## Status: Research
## Date: 2026-02-16
## Related: `decouple-templates.md`, `versioned-settings-refactor.md`

---

## 1. Executive Summary

This document explores the feasibility, challenges, and approaches to transforming Scion's template system from a harness-coupled model (each template is 1:1 with a harness) to a harness-agnostic model where a template defines the *role* of an agent (system prompt, skills, resources) and is combined with a harness-config at creation time.

The refactor is feasible but touches many layers of the system. The key challenges are: (1) separating harness-specific home directory content from portable template content, (2) rethinking how embedded defaults work, (3) adapting the provisioning flow to a two-phase "compose template + apply harness" model, and (4) deciding where harness-specific home directory files live (in harness-configs, in a global base layer, or generated by harness code).

Three approaches are evaluated. The recommended path is **Approach B: Harness Base Layers + Agnostic Templates**, which introduces a harness-owned "base home" layer that is composed with the harness-agnostic template at agent creation time.

## 2. Current State Analysis

### 2.1 The Template ↔ Harness Coupling

Today, templates and harnesses are tightly coupled at every level:

**Source embeds** (`pkg/config/embeds/`): Each harness has its own embed directory containing a complete agent home layout. There is no shared "template" concept that spans harnesses.

```
pkg/config/embeds/
├── common/                    # Only .tmux.conf is truly shared
│   └── .tmux.conf
├── claude/                    # Claude-coupled template
│   ├── scion-agent.yaml       # declares harness: claude
│   ├── .claude.json           # Claude Code state file
│   ├── settings.json          # Claude Code settings (hooks, env, permissions)
│   ├── claude.md              # Scion-specific instructions (sciontool hooks)
│   └── bashrc                 # alias g=gemini (!)
├── gemini/                    # Gemini-coupled template
│   ├── scion-agent.yaml       # declares harness: gemini + auth config
│   ├── settings.json          # Gemini CLI settings (hooks, model, security)
│   ├── gemini.md              # Scion-specific instructions (sciontool hooks)
│   ├── system_prompt.md       # Placeholder
│   └── bashrc                 # alias g=gemini
├── codex/                     # Codex-coupled template
│   ├── scion-agent.yaml       # declares harness: codex
│   ├── config.toml            # Codex config (model, approval policy)
│   └── bashrc                 # alias c=codex
└── opencode/                  # OpenCode-coupled template
    ├── scion-agent.yaml       # declares harness: opencode
    └── opencode.json          # OpenCode config (theme)
```

**Template seeding** (`Harness.SeedTemplateDir()`): Each harness implementation controls its own template directory layout. The `SeedCommonFiles()` function in `pkg/config/init.go` provides minimal shared infrastructure but the harness drives the process.

**Template discovery** (`pkg/config/templates.go`): Templates are discovered by name (e.g., `gemini`, `claude`) and their `scion-agent.yaml` declares which harness they bind to. The template name conventionally matches the harness name.

**Agent provisioning** (`pkg/agent/provision.go`): The provisioning flow copies `template/home/` into `agent/home/`, then calls `harness.Provision()` to do harness-specific setup (e.g., Claude updates `.claude.json` with workspace paths).

### 2.2 What Is Actually Harness-Specific vs Portable

To determine feasibility, we must classify every piece of template content:

#### Harness-Specific Content (must remain with harness)

| Content | Harness | Purpose |
|---------|---------|---------|
| `.claude.json` | Claude | CLI state file (onboarding, project trust, MCP servers) |
| `.claude/settings.json` | Claude | Claude Code settings (env vars, hooks, permissions, auto-updater) |
| `.gemini/settings.json` | Gemini | Gemini CLI settings (hooks, model, yolo mode, auth, security) |
| `.codex/config.toml` | Codex | Codex config (model, approval policy, workspace trust) |
| `.config/opencode/opencode.json` | OpenCode | OpenCode config (theme) |
| Auth discovery logic | All | Each harness finds credentials differently |
| Container command | All | Each harness has different CLI invocation (`gemini --yolo`, `claude --no-chrome`, etc.) |
| Interrupt key | All | Claude uses Escape, others use C-c |
| Hook dialect | All | `sciontool hook --dialect=claude` vs `--dialect=gemini` |

#### Portable Content (should belong to template)

| Content | Current Location | Purpose |
|---------|-----------------|---------|
| System prompt | `.claude/claude.md`, `.gemini/system_prompt.md` | Agent role definition |
| `.bashrc` | Per-harness embeds | Shell aliases (currently harness-specific but could be generic) |
| `.tmux.conf` | `common/` | Terminal multiplexer config (already shared) |
| `scion-agent.yaml` fields: `env`, `volumes`, `resources`, `services`, `max_turns`, `max_duration` | Template | Agent runtime configuration |

#### Ambiguous Content

| Content | Issue |
|---------|-------|
| `claude.md` / `gemini.md` | Currently contain identical Scion-specific instructions (sciontool hooks). The content is portable but the *delivery mechanism* differs (Claude reads `.claude/claude.md`, Gemini reads `.gemini/gemini.md`) |
| `.bashrc` | Currently contains harness-specific aliases (`alias g=gemini`). Could be split into portable + harness-specific |
| Hooks configuration in `settings.json` | The sciontool hook integration (`sciontool hook --dialect=X`) is functionally identical across harnesses but the settings format and hook event names differ |

### 2.3 Key Observation

The current `claude.md` and `gemini.md` files are **identical** — they both contain the same sciontool status-reporting instructions. This validates the premise that templates should be harness-agnostic: the actual agent instructions are already portable, only the file location and delivery format differ.

## 3. What a Harness-Agnostic Template Looks Like

### 3.1 Target Template Structure

A harness-agnostic template defines the agent's *purpose*, not its *execution mechanics*:

```yaml
# .scion/templates/code-reviewer/scion-template.yaml
schema_version: "1"
name: code-reviewer
description: "Thorough code review agent with security focus"

# Portable agent configuration
system_prompt: system-prompt.md    # path relative to template dir

env:
  REVIEW_STRICTNESS: high

resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "2"
    memory: "4Gi"

max_turns: 100
max_duration: "4h"

services:
  - name: browser
    command: ["chromium", "--headless"]
    ready_check:
      type: tcp
      target: "localhost:9222"
```

```
.scion/templates/code-reviewer/
├── scion-template.yaml          # Template metadata (harness-agnostic)
├── system-prompt.md             # Portable system prompt
└── home/                        # Portable home directory content (optional)
    └── .config/
        └── lint-rules/          # Custom config files
```

### 3.2 How It Combines with Harness-Config

At `scion create`, the user specifies both template and harness-config:

```bash
scion create my-agent --template code-reviewer --harness-config gemini
```

The system resolves:
1. **Template** → `code-reviewer` (provides system prompt, env, resources, services)
2. **Harness-config** → `gemini` from `settings.yaml` (provides image, user, harness type)
3. **Harness** → `gemini` (provides home dir setup, auth, command, hooks)

The agent home directory is assembled by composing:
1. Harness base layer (harness-specific config files)
2. Template home content (portable files)
3. System prompt injection (harness-specific delivery)

## 4. Challenges

### 4.1 Harness-Specific Home Directory Files

**The core challenge.** Each harness requires specific files in the agent home directory:

- Claude needs `.claude.json` and `.claude/settings.json`
- Gemini needs `.gemini/settings.json`
- Codex needs `.codex/config.toml`
- OpenCode needs `.config/opencode/opencode.json`

These files are currently embedded in templates. In a harness-agnostic world, these files must come from somewhere else. The question is: where?

Options:
- **A) Generated by `Harness.Provision()`**: The harness creates these files at provisioning time from embedded defaults. This moves file creation from template seeding to agent creation.
- **B) Stored in a "harness base" directory**: A separate `~/.scion/harness-bases/<harness>/home/` directory holds harness-specific files, composed with the template at creation time.
- **C) Stored in harness-config entries**: The `harness_configs` in `settings.yaml` gain a `home_dir` field pointing to harness-specific files.

### 4.2 System Prompt Delivery

Each harness reads system prompts from different locations:
- Claude: `.claude/claude.md` (read by Claude Code as CLAUDE.md equivalent)
- Gemini: `.gemini/system_prompt.md` (read by Gemini CLI)
- Codex: No system prompt mechanism currently
- OpenCode: No system prompt mechanism currently

A harness-agnostic template has a single `system-prompt.md`. The harness must know how to deliver it to the right location. This is a **transform** operation: take portable content and place it where the harness expects it.

### 4.3 Settings/Hooks Configuration

The `settings.json` files for Claude and Gemini contain critical integration hooks:

**Claude** (`settings.json`):
```json
{
  "hooks": {
    "SessionStart": [{"command": "sciontool hook --dialect=claude"}],
    "PostToolUse": [{"command": "sciontool hook --dialect=claude"}],
    ...
  },
  "env": { "CLAUDE_CODE_USE_VERTEX": "1", ... },
  "permissions": { "allow": ["*"] }
}
```

**Gemini** (`settings.json`):
```json
{
  "hooks": {
    "SessionStart": [{"command": "sciontool hook --dialect=gemini"}],
    "BeforeAgent": [{"command": "sciontool hook --dialect=gemini"}],
    ...
  },
  "yolo": true,
  "model": { "name": "gemini-3-flash-preview" }
}
```

These are harness-specific in format and content. They cannot be made portable. They must be generated or provided by the harness layer.

### 4.4 SeedTemplateDir Inversion

Currently the harness controls template creation via `SeedTemplateDir()`. With harness-agnostic templates, there are two distinct operations:

1. **Template creation**: Creates the portable template structure (system prompt, env, resources). This is harness-independent.
2. **Harness base setup**: Creates the harness-specific home directory files. This is template-independent.

The current `SeedTemplateDir()` conflates both. It must be split.

### 4.5 `scion-agent.yaml` vs `scion-template.yaml`

Currently `scion-agent.yaml` in the template directory declares both the harness binding (`harness: claude`) and the agent configuration (env, volumes, etc.). In the new model:

- `scion-template.yaml` replaces `scion-agent.yaml` in harness-agnostic templates
- It contains no `harness` field
- The harness binding comes from `--harness-config` at creation time

For backward compatibility, legacy templates with `scion-agent.yaml` + `harness` field continue to work as before.

### 4.6 Embed Restructuring

The current `pkg/config/embeds/` directory is organized by harness. In a harness-agnostic world, we need to separate:

- **Harness base files**: Still organized by harness name (`.claude.json`, `settings.json`, etc.)
- **Default templates**: Harness-agnostic templates that ship with the binary
- **Common files**: Shared across all (`.tmux.conf`, etc.)

### 4.7 Harness.Provision() Must Handle Missing Files

Currently `Provision()` assumes the template has already placed harness-specific files (e.g., Claude's `Provision()` reads an existing `.claude.json` to update workspace paths). In the new model, `Provision()` must handle the case where these files don't exist yet and create them.

### 4.8 Hub Template Storage

In hosted mode, templates are stored in the Hub. Harness-agnostic templates would be a different format/structure than legacy templates. The Hub API and storage would need to support both, or at minimum the new format.

## 5. Approaches

### Approach A: Template Composition with Adapter Directories

**Already explored in `decouple-templates.md`.** Each template contains a `base/` directory and per-harness `adapters/<harness>/` directories.

```
code-reviewer/
├── scion-template.yaml
├── base/
│   ├── home/
│   │   └── .config/...
│   └── system-prompt.md
└── adapters/
    ├── claude/
    │   ├── home/.claude/...
    │   └── adapter.yaml
    └── gemini/
        ├── home/.gemini/...
        └── adapter.yaml
```

**Pros:**
- Self-contained: everything needed is in the template directory
- Explicit harness compatibility declared via adapter presence
- Transform system allows deriving harness-specific files from base content

**Cons:**
- Duplicates harness-specific files across every template
- Each new template must include adapters for every harness it supports
- Maintaining adapters becomes a burden — updating Claude's `settings.json` format requires updating it in every template
- Template authors need to understand harness internals to write adapters

**Verdict:** Attractive in theory but scales poorly. The adapter content (`.claude.json`, `settings.json`, hooks config) is effectively identical across all templates. Duplicating it in every template creates a maintenance burden.

### Approach B: Harness Base Layers + Agnostic Templates (Recommended)

Separate the harness-specific home directory content into **harness base layers** that are independent of templates. Templates contain only portable content. At agent creation, the system composes: harness base + template overlay.

#### Harness Base Layers

Each harness has a "base home" directory containing its required configuration files. These are seeded during `scion init` and stored in the grove/global `.scion/` directory:

```
~/.scion/
├── harness-homes/             # NEW: harness base layers
│   ├── claude/
│   │   ├── .claude.json
│   │   ├── .claude/
│   │   │   └── settings.json
│   │   └── .bashrc
│   ├── gemini/
│   │   ├── .gemini/
│   │   │   └── settings.json
│   │   └── .bashrc
│   ├── codex/
│   │   ├── .codex/
│   │   │   └── config.toml
│   │   └── .bashrc
│   └── opencode/
│       ├── .config/
│       │   └── opencode/
│       │       └── opencode.json
│       └── .bashrc
├── templates/                 # NOW: harness-agnostic templates
│   ├── code-reviewer/
│   │   ├── scion-template.yaml
│   │   └── system-prompt.md
│   ├── researcher/
│   │   ├── scion-template.yaml
│   │   └── system-prompt.md
│   └── gemini/                # Legacy coupled template (still works)
│       ├── scion-agent.yaml
│       └── home/
│           └── .gemini/...
└── settings.yaml
```

#### Composition at Agent Creation

```
Agent Home = Harness Base Layer
           + Template Home (if any)
           + System Prompt Injection
           + Settings/Profile Overrides
           + Common Files (.tmux.conf)
```

Steps:
1. Copy harness base home → agent home
2. Copy template home → agent home (overlay, template files win on conflict)
3. Inject system prompt into harness-specific location
4. Copy common files (.tmux.conf)
5. Call `Harness.Provision()` for dynamic setup (e.g., workspace paths in `.claude.json`)

#### System Prompt Injection

The harness interface gains a new method:

```go
type Harness interface {
    // ... existing methods ...

    // InjectSystemPrompt writes the system prompt content to the
    // harness-specific location in the agent home directory.
    InjectSystemPrompt(agentHome string, content []byte) error
}
```

Implementations:
- **Claude**: Writes to `{agentHome}/.claude/CLAUDE.md`
- **Gemini**: Writes to `{agentHome}/.gemini/system_prompt.md`
- **Codex**: No-op (or future: write to `.codex/instructions.md`)
- **OpenCode**: No-op (or future: writes config section)
- **Generic**: Writes to `{agentHome}/.scion/system_prompt.md`

#### Embed Restructuring

```
pkg/config/embeds/
├── common/
│   └── .tmux.conf
├── harness-homes/              # NEW: harness base home files
│   ├── claude/
│   │   ├── .claude.json
│   │   ├── settings.json       # Goes to .claude/settings.json
│   │   └── bashrc
│   ├── gemini/
│   │   ├── settings.json       # Goes to .gemini/settings.json
│   │   └── bashrc
│   ├── codex/
│   │   ├── config.toml
│   │   └── bashrc
│   └── opencode/
│       └── opencode.json
├── templates/                  # NEW: default agnostic templates
│   ├── default/
│   │   ├── scion-template.yaml
│   │   └── system-prompt.md
│   └── (future: code-reviewer/, researcher/, etc.)
├── legacy-templates/           # RENAMED from claude/, gemini/, etc.
│   ├── claude/
│   │   └── scion-agent.yaml
│   ├── gemini/
│   │   └── scion-agent.yaml
│   ├── codex/
│   │   └── scion-agent.yaml
│   └── opencode/
│       └── scion-agent.yaml
└── default_settings.yaml
```

**Pros:**
- Harness-specific content maintained once, not duplicated per template
- Templates are purely about agent purpose — easy to author, share, version
- Clean separation of concerns
- Harness base files can be updated independently (e.g., `scion update --harness-homes`)
- Backward compatible: legacy templates still work via existing detection

**Cons:**
- Two new concepts for users to understand: harness-homes + agnostic templates
- Extra directory to manage in `.scion/`
- Composition logic adds complexity to provisioning

### Approach C: Harness-Generated Home (No Persistent Base Layer)

Instead of persisting harness base files on disk, the harness generates them on-the-fly during provisioning. The `Provision()` method becomes the sole source of harness-specific files.

```go
// Harness.Provision() now creates all harness-specific files
func (c *ClaudeCode) Provision(ctx context.Context, agentName, agentHome, agentWorkspace string) error {
    // 1. Create .claude.json from embedded defaults
    claudeJSON := generateClaudeJSON(agentWorkspace)
    os.WriteFile(filepath.Join(agentHome, ".claude.json"), claudeJSON, 0644)

    // 2. Create .claude/settings.json from embedded defaults
    settings := generateClaudeSettings()
    os.WriteFile(filepath.Join(agentHome, ".claude", "settings.json"), settings, 0644)

    // 3. Create .bashrc
    os.WriteFile(filepath.Join(agentHome, ".bashrc"), bashrcContent, 0644)

    return nil
}
```

#### Template Interaction

Templates provide portable content. The provisioning flow becomes:

1. Copy template home → agent home
2. Call `Harness.SetupHome(agentHome)` — creates harness-specific files
3. Call `Harness.InjectSystemPrompt(agentHome, prompt)` — delivers system prompt
4. Call `Harness.Provision(agentHome, workspace)` — dynamic setup (workspace paths)

**Pros:**
- Simplest model: no new directories, no harness-homes to manage
- Harness code is the single source of truth for its files
- Users never need to know about harness base files
- Updating harness defaults = updating Go code, automatically picked up

**Cons:**
- Harness-specific files are not user-customizable (they're generated from Go code, not disk files)
- Users can't customize the Claude `settings.json` hooks or Gemini `settings.json` model without post-provisioning edits
- `SeedTemplateDir()` for legacy templates would duplicate this generation logic
- Breaks the current model where users can edit template files to customize harness behavior

## 6. Detailed Comparison

| Aspect | A: Adapters in Template | B: Harness Base Layers | C: Generated |
|--------|------------------------|----------------------|-------------|
| Template authoring complexity | High (must write adapters) | Low (portable only) | Low (portable only) |
| Harness file duplication | Per-template duplication | Single copy per grove | None (in code) |
| User customizability | Per-template customization | Per-grove customization | Not customizable |
| Maintenance burden | High (N templates × M harnesses) | Low (M harness bases) | Low (in code) |
| Disk footprint | Large | Moderate | Minimal |
| Backward compatibility | Good (detect by scion-template.yaml) | Good | Fair (Provision() changes) |
| New template creation | Must create adapters | Just write portable content | Just write portable content |
| Updating harness defaults | Update in every template | Update once in harness-homes | Update Go code |
| Sharing templates | Self-contained but large | Small, portable | Small, portable |

## 7. Implementation Sketch (Approach B)

### 7.1 Phase 1: Harness Base Layer Infrastructure

**Goal:** Create the harness-homes mechanism and `InjectSystemPrompt` without changing any existing behavior.

1. Create `pkg/config/embeds/harness-homes/` by extracting harness-specific home files from current per-harness embed directories.

2. Add `SeedHarnessHome(harnessName, targetDir string, force bool)` to `pkg/config/init.go` — seeds harness base files into `~/.scion/harness-homes/<harness>/` or `.scion/harness-homes/<harness>/`.

3. Update `InitProject()` and `InitGlobal()` to seed harness-homes alongside templates.

4. Add `InjectSystemPrompt(agentHome string, content []byte) error` to the `Harness` interface.

5. Implement `InjectSystemPrompt` for each harness:
   - Claude: write to `.claude/CLAUDE.md`
   - Gemini: write to `.gemini/system_prompt.md`
   - Generic: write to `.scion/system_prompt.md`

### 7.2 Phase 2: Agnostic Template Support

**Goal:** Support `scion-template.yaml` as a new template format that is harness-agnostic.

1. Define `scion-template.yaml` schema:
   ```yaml
   schema_version: "1"
   name: string
   description: string
   system_prompt: string     # relative path to system prompt file
   env: map[string]string
   volumes: []VolumeMount
   resources: ResourceSpec
   max_turns: int
   max_duration: string
   services: []ServiceSpec
   ```

2. Update `Template.LoadConfig()` to detect and load `scion-template.yaml`.

3. Add `Template.IsAgnostic() bool` — returns true if template has `scion-template.yaml` without a `harness` field.

4. Update template discovery to handle agnostic templates.

### 7.3 Phase 3: Composition in Provisioning

**Goal:** Wire the composition flow into `ProvisionAgent()`.

1. In `ProvisionAgent()`, after template detection:
   - If agnostic template: compose harness-home + template
   - If legacy template: use existing flow

2. Composition flow:
   ```go
   func composeAgentHome(harnessName, agentHome, templateDir string) error {
       // 1. Copy harness base home
       harnessHome := findHarnessHome(harnessName, grovePath)
       util.CopyDir(harnessHome, agentHome)

       // 2. Copy template home (overlay)
       templateHome := filepath.Join(templateDir, "home")
       if exists(templateHome) {
           util.CopyDirMerge(templateHome, agentHome)
       }

       // 3. Copy common files
       copyCommonFiles(agentHome)  // .tmux.conf

       // 4. Inject system prompt
       h := harness.New(harnessName)
       if promptPath := template.SystemPromptPath(); promptPath != "" {
           content, _ := os.ReadFile(promptPath)
           h.InjectSystemPrompt(agentHome, content)
       }

       return nil
   }
   ```

3. Require harness-config to be specified for agnostic templates:
   - `--harness-config` flag on CLI
   - Profile `default_harness_config` as fallback
   - Error if neither is provided and template is agnostic

### 7.4 Phase 4: Embed and Init Restructuring

**Goal:** Restructure embeds and update seeding logic.

1. Move harness-specific embed files from `embeds/<harness>/` to `embeds/harness-homes/<harness>/`.

2. Keep `embeds/<harness>/scion-agent.yaml` as legacy template config.

3. Add default agnostic template(s) to `embeds/templates/`.

4. Update `SeedTemplateDir()` implementations to use the new locations.

5. Add `SeedHarnessHome()` calls to `InitProject()` and `InitGlobal()`.

### 7.5 Phase 5: Default Template Migration

**Goal:** Ship default agnostic templates alongside legacy ones.

1. Create a `default` agnostic template with a generic system prompt.

2. Legacy templates (`claude`, `gemini`, `codex`, `opencode`) remain as-is for backward compatibility.

3. New templates can be created as agnostic: `scion template create my-template`.

4. Future: `scion template decouple gemini --name my-gemini-template` to extract portable content from a legacy template.

## 8. Key Design Decisions Required

### 8.1 How to handle `scion-agent.yaml` in agnostic templates?

**Option 1:** Replace with `scion-template.yaml` — a new file format without a `harness` field. Detection: presence of `scion-template.yaml` = agnostic, presence of `scion-agent.yaml` with `harness` field = legacy.

**Option 2:** Keep `scion-agent.yaml` but make `harness` field optional. If missing, the template is agnostic.

**Recommendation:** Option 1. A new file name (`scion-template.yaml`) makes the distinction explicit and avoids ambiguity. The `scion-agent.yaml` continues to work for legacy templates.

### 8.2 Where do harness-homes live?

**Option 1:** `~/.scion/harness-homes/<harness>/` (global) and `.scion/harness-homes/<harness>/` (grove-level override).

**Option 2:** Only global `~/.scion/harness-homes/<harness>/`. No grove-level override.

**Option 3:** Not on disk at all — generated by harness code at provisioning time (Approach C).

**Recommendation:** Option 1, with grove-level overrides for flexibility. Most users only need the global version. Power users can customize per-grove.

### 8.3 Should harness-configs include home directory content?

Currently `HarnessConfigEntry` has image, user, model, args, env, volumes, auth. Should it also specify harness home content?

**Recommendation:** No. Keep harness-configs focused on runtime parameters (image, user, env). The harness home content is more stable and should be managed separately. Mixing home directory files into harness-configs would make the settings.yaml too complex.

### 8.4 How does a user customize harness home files?

With Approach B, users edit files in `~/.scion/harness-homes/<harness>/`. For example, to add custom hooks to Claude's settings.json, they edit `~/.scion/harness-homes/claude/.claude/settings.json`.

There should also be a reset mechanism: `scion harness reset claude` to restore defaults from embedded files.

### 8.5 What happens during `scion init` for harness-homes?

`scion init` should seed harness-homes for all known harnesses, just as it currently seeds templates. If harness-homes already exist, they are not overwritten (unless `--force`).

### 8.6 How do legacy coupled templates interact with harness-homes?

Legacy templates (with `scion-agent.yaml` declaring `harness: X`) continue to use the existing flow — they contain their own home directory with harness-specific files. Harness-homes are only used for agnostic templates.

This means during transition, both mechanisms coexist. Legacy templates are self-contained; agnostic templates compose from harness-homes.

### 8.7 What about the `harness` field in `ScionConfig`?

For agnostic templates, the `harness` field in the resulting `ScionConfig` is derived from the harness-config entry, not from the template. The provisioning code resolves:

```
harness = harness_configs[harness_config_name].harness
```

The template's `scion-template.yaml` does not need a harness field.

## 10. Migration Path

### Step 1: No Breaking Changes (Foundation)
- Add `InjectSystemPrompt()` to harness interface (no-op default)
- Add harness-homes infrastructure (seeding, discovery)
- Keep all existing templates working exactly as before

### Step 2: Agnostic Template Support (Additive)
- Support `scion-template.yaml` detection
- Implement composition in provisioning (harness-home + template)
- Add `scion template create` for agnostic templates
- Legacy templates continue to work

### Step 3: Default Template Evolution (Opt-in)
- Ship default agnostic templates alongside legacy ones
- Users can choose to use agnostic templates for new agents
- `scion template decouple` for migrating existing templates

### Step 4: (Future, Optional) Legacy Deprecation
- Emit warnings for legacy coupled templates
- Guide users to `scion template decouple`
- Only after agnostic templates are battle-tested

## 11. Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Composition order bugs (harness base vs template vs common) | Files in wrong location or overwritten | Comprehensive tests with all harness × template combinations |
| System prompt injection fails for new harnesses | Agent runs without instructions | `InjectSystemPrompt` returns error; provisioning fails early |
| Users customize harness-homes then `scion init --force` overwrites them | Lost customizations | Backup before overwrite, add `--no-harness-homes` flag |
| Agnostic template + missing harness-config = confusing error | User doesn't understand what to specify | Clear error message: "Template 'X' is harness-agnostic. Specify --harness-config or set default_harness_config in your profile." |
| Hub storage format diverges for agnostic vs legacy templates | Complex Hub API | Unified storage format that handles both; Hub detects template type |
| Harness base files drift from embedded defaults | Stale config | `scion harness check` to compare on-disk files with embedded defaults |

## 12. Recommendation

**Implement Approach B (Harness Base Layers + Agnostic Templates)** in phases, starting with the foundation (harness-homes, InjectSystemPrompt) and adding agnostic template support as an additive feature.

This approach:
- Cleanly separates concerns (template = purpose, harness = mechanics)
- Avoids the per-template duplication problem of Approach A
- Preserves user customizability (unlike Approach C)
- Is fully backward compatible with existing coupled templates
- Builds on the existing `harness_configs` system from the versioned settings refactor
- Naturally extends the existing harness interface without breaking it

The key insight enabling this approach: harness-specific home directory content is **identical across all templates** using the same harness. There is no reason to duplicate `.claude.json` in every Claude template. One harness base layer per harness is sufficient.
